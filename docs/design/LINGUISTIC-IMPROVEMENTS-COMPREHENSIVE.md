## Overview

This document provides linguistic improvements for all tool interfaces in the structs-mcp project, making them more human-friendly and easier for AI agents to understand and use correctly.

---

## Common Patterns to Fix

### 1. **Implementation Details in Descriptions**
- ‚ùå "from consensus API"
- ‚ùå "from database"
- ‚ùå "using signer.tx_* functions"
- ‚ùå "database handles signing"
- ‚ùå "requires DATABASE_URL"

### 2. **Technical Jargon Without Context**
- ‚ùå "operate_ambit" (what is an ambit?)
- ‚ùå "struct_type_id" (what types are available?)
- ‚ùå "consensus API" (what is this?)
- ‚ùå "pagination key" (what does this mean?)

### 3. **Tool Name References**
- ‚ùå "returned by structs_workflow_execute"
- ‚ùå "started by structs_calculate_proof_of_work"

### 4. **Conditional/Contextual Descriptions**
- ‚ùå "Entity ID (struct ID for building/mining/refining, fleet ID for raids)"
- ‚ùå Parameters that vary by action type without clear explanation

### 5. **Unclear Action Names**
- ‚ùå "struct_build_initiate" vs "struct_build_complete"
- ‚ùå Technical message names instead of user-friendly actions

## Summary of Principles

1. **Remove Implementation Details**: No mentions of "consensus API", "database", "signer.tx_*", internal function names
2. **Use Natural Language**: "Get information" vs "Query state", "Check if" vs "Validate"
3. **Explain Technical Terms**: Define "ambit", "slot", "struct_type_id" in context
4. **Focus on User Intent**: What does the user want to accomplish?
5. **Provide Context**: Why would you use this? When is it optional?
6. **Avoid Tool Name References**: Don't reference other tools by name in descriptions
7. **Add Examples**: Show what values look like, what they mean
8. **Explain Benefits**: Why would you use this tool? What problem does it solve?
9. **Use Conversational Tone**: "You", "your", "I'll" where appropriate
10. **Clarify Conditional Logic**: When parameters vary by action, explain clearly
